## Part 1: The Journey

### Where you started
A static HTML site. One `.html` file per page. CSS rules copy-pasted across pages. To change a button color, you had to change it in 23 files and hope you didn't miss any. Designers fought with hand-coded `<style>` blocks. Content and code were tangled together.

### Where you are now
A **content-driven static site generated by Astro**. The browser still sees plain HTML/CSS — but the *process* of producing it is different:

- Content lives in **markdown files** (`.md`) like blog posts
- Code lives in **Astro components** (`.astro`)
- Astro reads everything, validates it against a schema, combines it, and outputs clean static HTML
- The output is a `dist/` folder with 24 HTML files. Drop that folder on any web server. No database, no Node.js in production, no PHP. Just files.

### The journey

**Result:** A 24-page site that builds in 5 seconds, has zero duplicated CSS, and is ready to deploy.

## Part 2: Your Intuition (the WordPress-to-Astro map)

You have a WordPress background. You think in **CPTs** (Custom Post Types), **field groups**, and **meta fields**. Astro has analogs. Here's the map:

| WordPress concept | Astro equivalent | Where it lives in this project |
|---|---|---|
| **CPT** (Custom Post Type) | **Content collection** | `src/content/menus/` |
| **Field group** | **Zod schema** | `src/content.config.ts` |
| **Meta fields** (post meta) | **YAML frontmatter** at the top of each `.md` file | At the top of every `*.md` file |
| **Template** (single.php, page.php) | **Astro component** | `src/components/MenuSection.astro`, `MenuPackage.astro` |
| **The Loop** | A `for` loop in the page template | `data.sections.map((section) => <MenuSection section={section} />)` |
| **wp-admin editor** | A text editor + Git | Your code editor, or GitHub's web editor |
| **Taxonomy** (categories, tags) | Frontmatter fields with `z.enum([...])` | `service: z.enum(['lunch', 'dinner', 'banquet', ...])` |
| **Custom field UI** (ACF, Metabox) | None — the schema IS the UI. If you forget a field, the build fails with a clear error. | N/A — the schema enforces the shape |
| **Page builder** (Elementor, etc.) | Astro components with `<slot />` | `BaseLayout.astro` has a `<slot />` where each page's content goes |

The big shift: in WordPress, content lives in a database. In Astro, content lives in **files in your repo**. The file system is the database. Git is the version history. The schema (in `content.config.ts`) is the validation layer.

## Part 3: Astro as a Concept (30,000 ft)

Astro is a **static site generator**. Three ideas to understand:

### Idea 1: Components are islands
In a typical React/Vue app, the entire page is one big running JavaScript program. In Astro, the page is **HTML by default**. You can sprinkle in interactive components ("islands"), but most of the page is just static HTML. This is why Astro sites load fast — there's no JS framework runtime cost.

### Idea 2: Content collections are typed
Astro has a built-in system for "this folder contains blog posts" or "this folder contains menu items." You write a **schema** that says what fields each item must have (`title: string, price: number, sections: array`). Astro validates every file at build time. If a required field is missing, the build fails — you can't ship broken content.

### Idea 3: The build is the deploy
When you run `npm run build`, Astro reads all your source files and produces a `dist/` folder. That folder is your website. No server needed. No Node.js in production. Just static files served by any web host.

## Part 4: The Content System (CPTs explained for real)

### The schema: `src/content.config.ts`

This is your **field group**. It defines what every menu file MUST look like.

```ts
const menuSection = z.object({
  heading: z.string(),                                // required: text
  layout: z.enum(['items', 'sides', 'pricelist',     // required: dropdown
                  'wine', 'info', 'promo'])
                  .default('items'),
  items: z.array(menuItem).optional(),                // optional: list
  columns: z.array(...).optional(),                    // optional: list
});
```

- `z.object({...})` means "an object with these fields"
- `z.string()` means "must be text"
- `z.enum([...])` means "must be one of these values"
- `.optional()` means "may not be there"
- `.default(...)` means "if not specified, use this"

**This is your rulebook.** Every menu file gets checked against it. If a section is missing a heading, the build fails with: `Expected string, received undefined`. If a `layout` value is misspelled, the build fails with: `Expected one of items|sides|pricelist|wine|info|promo, got "itemss"`.

You can never ship broken content because the build won't let you.

### The content: `src/content/menus/*.md`

This is your **CPT collection**. Each `.md` file is one "post."

```markdown
---
title: Dinner Menu
service: dinner
priceTier: $$$
sections:
  - heading: Appetizers
    layout: items
    items:
      - name: Wings
        price: "$12"
---

# Free-form markdown here
```

The block between the `---` markers is the **frontmatter** (think: post meta fields). The body is the post content. The frontmatter gets validated against your schema at build time.

### The render: `MenuSection.astro`

This is your **template**. It receives one section of data and decides how to render it based on `layout`:

```astro
{layout === 'items' && <ItemsLayout items={items} />}
{layout === 'pricelist' && <PricelistLayout columns={columns} />}
{layout === 'wine' && <WineLayout house={house} groups={glassGroups} />}
```

Same component, six different layouts, one source of truth for the design.

## Part 5: The Component System (the magic explained)

### How `MenuSection.astro` works

The component is a function that takes a section of data and returns HTML. The "magic" is that it decides which layout to render based on the `layout` field. Here's the journey:

1. **Page** (`dinner-menu.astro`) calls `getEntry('menus', 'dinner')` — this reads the markdown file
2. The data has a `sections` array
3. For each section, the page renders `<MenuSection section={section} />`
4. `MenuSection` looks at `section.layout` and renders the right HTML
5. Astro's scoped CSS adds a unique hash to all the class names
6. The final HTML is written to `dist/dinner-menu/index.html`

**Same component, six layouts, one source of truth.** If you want to change how "wine" sections look, you change one file (`MenuSection.astro`). All pages using that layout get the new look automatically.

### Why scoped CSS is the secret sauce

In normal CSS, you write `.sp-menu__item` and it applies everywhere. If two pages use the same class name for different things, you get conflicts. CSS specificity wars begin.

In Astro, `<style>` blocks are **scoped** to the component. When you write `.sp-menu__item` in `MenuSection.astro`, Astro adds a unique attribute to both the CSS rule AND the rendered HTML. So it becomes:

```css
.sp-menu__item[data-astro-cid-abc123] { ... }
```

Only elements rendered BY `MenuSection.astro` get that attribute, so only those elements match the rule.

This is why you can have `.sp-pkg__item` in `MenuSection.astro` and a different `.sp-pkg__item` in `MenuPackage.astro` and they don't conflict. Each component owns its own scope.

### The `data` journey — start to finish

Here's what happens when you run `npm run build`:

1. Astro reads `src/content.config.ts` to learn the schema
2. Astro scans `src/content/menus/*.md` and validates each against the schema
3. If validation fails, the build fails with a clear error
4. For each page route, Astro runs the page component
5. The page calls `getEntry('menus', 'dinner')` which returns the validated data
6. The page iterates over `data.sections` and calls `MenuSection` for each
7. `MenuSection` returns HTML based on the layout
8. Astro's scoped CSS adds the `data-astro-cid-xxx` attributes
9. The final HTML is written to `dist/dinner-menu/index.html`

**This whole process is the transport:** schema → content → component → HTML. Everything is type-checked at every step.

## Part 6: The Quality-of-Life Magic (why devs love Astro)

### 1. Image optimization (built-in, you haven't used it yet)
Drop a `.jpg` in `public/img/` and reference it. With the `<Image>` component, Astro can:
- Resize to multiple sizes for responsive loading
- Convert to modern formats (WebP, AVIF) automatically
- Lazy-load below-the-fold images
- Inline critical above-the-fold images

**You don't configure any of this.** The build does it. Drop the image in, get the optimization out.

### 2. Zero JS by default
Most pages don't need JavaScript. Astro doesn't add a JS runtime unless you ask. Compare to a typical WordPress page which loads jQuery + 20 plugins even on a static content page. Your Astro site ships only the HTML + CSS that's actually used. **Result: faster load, better mobile experience, better SEO.**

### 3. TypeScript everywhere
Your `content.config.ts` uses Zod to define the schema. When you write `data.sections[0].heading` in your component, TypeScript knows `heading` is a string. No manual type definitions needed — Astro generates them from your schema automatically. **You get autocomplete and error-checking in your editor for free.**

### 4. Hot module replacement during dev
When you run `npm run dev` and edit a file, the browser updates instantly. No full page refresh. CSS changes are reflected without losing form input. **Iteration speed goes up dramatically.**

### 5. Content collections are type-safe
`getEntry('menus', 'dinner')` returns a fully typed object. If you try to access a field that doesn't exist, TypeScript yells at you. The build also fails if any content file is missing required fields. **You can't accidentally ship broken content.**

### 6. Component-scoped CSS
Already explained. The killer feature for component-based design. **No more naming wars, no more `is:global` leakage, no more debugging "why is this style applying over there?"**

### 7. Build-time validation
Run `npm run build`. If a content file is missing a required field, if a URL points to a missing page, if a component prop is the wrong type — the build fails with a clear error. **You can't ship broken content.**

### 8. MDX support (if you want it later)
Right now your content files are plain markdown. With MDX, you can embed Astro components inside markdown. Want a "Book Now" button inside a menu section? You can write it inline:
```mdx
<CallToAction href="/contact" label="Book this package" />
```

### 9. View transitions (built-in, off by default)
Want fancy page-to-page animations? Astro has built-in support. Add `<ViewTransitions />` to your `BaseLayout` and you get smooth crossfades automatically.

## Part 7: Alternative Data Sources (what else is out there)

You've discovered one approach: Astro's content collections. The data lives in `.md` files in your repo. This is great for small sites where the developer (you) controls everything.

Other options exist:

### 1. Local file CMS (what you have now) — the Git-as-CMS approach
- `.md` files in your repo
- Edited via IDE, Git, or GitHub's web editor
- No external dependencies, no vendor lock-in
- Git tracks every change automatically
- **Best for:** Small sites, developer-driven content, version-controlled content
- **Cons:** Non-developers need to learn markdown + git
- **Verdict:** Right choice for your current scope

### 2. Google Sheets as a CMS
You've used this before. The flow:
- Restaurant owner edits a Google Sheet
- Astro's build script reads the Sheet via API at build time
- Build outputs static pages
- Trigger: a webhook from the Sheet tells Astro to rebuild
- **Best for:** Non-technical content owners who already use Google Sheets
- **Pros:** Restaurant owner can edit without touching code. Real-time updates.
- **Cons:** Build pipeline complexity. Schema validation is harder (Sheets is loose with types). Need a deploy hook.

### 3. Sanity (headless CMS)
The "Stripe of content." A proper CMS with a Studio UI for non-developers.
- Define schema in Sanity's dashboard
- Editors use Sanity Studio to write content
- Astro fetches via Sanity's API at build time
- Sanity handles images, localization, drafts, real-time preview
- **Best for:** Mid-to-large sites with multiple content editors
- **Pros:** Real CMS with rich UI. Image CDN. Real-time preview. Multi-user.
- **Cons:** External service (cost, vendor lock-in). More setup. Build pipeline integration.

### 4. Notion as a CMS
- Restaurant owner writes content in Notion
- Astro's build script reads Notion via API
- Same pattern as Sheets, but Notion has better structure (databases, properties)
- **Pros:** Owners already use Notion. Rich content (text, images, embeds).
- **Cons:** Notion's API is complex. Schema is implicit in the database structure.

### 5. Astro DB (newer, in beta)
Astro's own lightweight database. SQL-based, lives in your repo, syncs to a hosted version.
- Define schema in TypeScript
- Use SQL queries to fetch content
- Local-first, but with cloud sync
- **Best for:** Apps that need real database queries but want static-site performance
- **Pros:** Best of both worlds: real database with type safety, but local-first
- **Cons:** Beta. Adds complexity. Overkill for a restaurant site.

### For your case: local files
Sean Patrick's is a small site, you control the content, and the simplicity of "edit markdown, push, deploy" is valuable. **If the restaurant owner wants to edit menus without you, look at Sanity.** Otherwise, stay with what you have.

## Appendix C: The 3 Concepts You Should Internalize

1. **Data, design, page are separate.** Data lives in `.md` files. Design lives in `.astro` components. Pages are thin glue. If you find yourself editing a page to change how something looks, you're probably doing it wrong — edit the component instead.

2. **The schema is the contract.** `src/content.config.ts` defines what content is valid. Every `.md` file must conform. If you want to add a new field, update the schema first, then the content files. The build enforces this.

3. **One source of truth, always.** If you find yourself copy-pasting CSS or markup between pages, stop. Extract a component. The 5-page pattern extraction eliminated ~1500 lines of duplication by making `MenuPackage.astro` the single source of truth. That's the pattern.
